## 1. User experience

### 1.1 Flow and coherence
- **Intended arc:** STATE analysis of transcript → options (B, F, E, D, A, H) → assessment closure (H) → user chose (c) bigger change → STATE updated. User requested relay (option 1) → system interpreted as "write to SCHOLAR" → user corrected: relay = seed for LEARN. LEARN session opened with seed → options (B, E, D, A, H) → synthesis (H) → follow-on (b) small incremental → user approved → SCHOLAR pattern note added. User requested SYSTEM analysis of STATE–SCHOLAR interaction → audit produced → user asked "shouldn't you ask me if i want to implement recommendation" → system offered 1/2/3/4 → user chose 1 (relay) → system wrote to SCHOLAR → user corrected relay meaning → system reverted SCHOLAR write and updated STATE log.
- **Assessment:** The arc was coherent once relay was redefined. The user drove the path (STATE → relay → LEARN → synthesis → small add to SCHOLAR) and got a clear outcome: STATE enriched (patterns, signal checks, log), LEARN explored the seed, SCHOLAR gained one pattern note with explicit approval.

### 1.2 Friction points
| Issue | What happened | Impact |
|-------|----------------|--------|
| **Options not automatic** | After first B response (power/structural), the 8 STATE options were not appended. User said "you are supposed to give me options automatically." | One correction; thereafter options were supplied after every substantive turn. |
| **Relay definition** | User said "relay" = use STATE session as seed to begin LEARN; system had implemented relay as "write session output into SCHOLAR." User: "when I say relay, I mean use the state session as a seed to begin the learn session; don't automatically write it into scholar." | System reverted the SCHOLAR block and updated STATE log to "Relay (seed for LEARN)… no automatic write to SCHOLAR." Clarification improved alignment with user intent. |
| **Recommendations not actionable** | After SYSTEM audit, system listed three recommendations but did not ask whether the user wanted to implement any. User: "shouldn't you ask me if i want to implement recommendation." | One correction; system then offered 1/2/3/4 and user chose 1 (relay). |

### 1.3 Positive experience
- **Options as menu:** User repeatedly chose from A–H (B, F, E, D, A, H in STATE; B, E, D, A, H in LEARN) and from (b)(c)(a) at closure. Options functioned as a capability menu.
- **Approval gates:** STATE (c) and LEARN (b) were applied only after user approval. SCHOLAR addition (b) was proposed as text and written only after "yes."
- **Explicit correction of relay:** User's definition (relay = seed for LEARN) is now recorded in STATE and can be adopted in governance/template so future sessions don’t auto-write to SCHOLAR on "relay."

## 3. Recommendations (procedural)

1. **Options menu:** In STATE and LEARN, append the full 8 options after **every** substantive response (including the first), without waiting for user to request them. Reduces need for user correction.
2. **Relay definition:** Update CIV–STATE–TEMPLATE §XIV-B and/or cursor rule cmc-state-scholar-harvest to distinguish (a) **relay = seed for LEARN** (STATE session content used to open a LEARN session; no direct write to SCHOLAR) from (b) **transfer to SCHOLAR** (after LEARN or explicit user request, propose and then write to SCHOLAR with approval). Or adopt the user’s definition as canonical: relay = seed for LEARN only; any SCHOLAR write is a separate step (e.g. "add to scholar" or follow-on from LEARN).
3. **SYSTEM recommendations:** When an audit or evaluation produces recommendations, end with an explicit offer: "Do you want to implement any of these? (1) … (2) … (3) … (4) None."
4. **STATE session start:** Enforce SCHOLAR load and MEM SCAN at STATE session start (or at first analytical node) per cmc-state-mem-grounding so STATE analysis is SCHOLAR- and MEM-grounded.

---

## 4. Recommendations (user experience)

1. **Confirm relay intent when "relay" is first used in a session:** e.g. "Do you want (a) this session used as a seed to start a LEARN session, or (b) this session’s output proposed for direct addition to SCHOLAR?" Avoids auto-write when user meant seed.
2. **Keep approval gates:** Continue to propose STATE/SCHOLAR edits as text and apply only after explicit user approval; same for "implement recommendation."
