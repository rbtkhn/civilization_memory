I. PURPOSE & AUTHORITY
────────────────────────────────────────────────────────────
This file defines the mandatory structure, evidentiary standards,
scholarly constraints, behavioral rules, and ingest behavior for all
MEM (Civilizational Memory) files within the Civilizational Memory Codex.

Core principle:
A MEM file must function as a self-contained historical intelligence
object—capable of activating structured inquiry even when ingested
alone into a new conversation.

MEM authoring is a jurisdictional act.

Authority Flow (NON-REVERSIBLE):
CIV–MEM–CORE → CIV–MEM–TEMPLATE → MEM Files → SCHOLAR / CORE / GAME / LLM

No MEM file is valid unless compliant with this template AND the
applicable ARC–[CIV] canon at the time of locking.

────────────────────────────────────────────────────────────
II. FILE IDENTITY & METADATA (MANDATORY)
────────────────────────────────────────────────────────────
Every MEM file MUST begin with the following metadata block,
in this exact order:

• Filename (canonical)
• Repository reference
• Status (ACTIVE / LOCKED / DRAFT)
• Version (semantic)
• Upgrade Type (if applicable)
• Civilization
• Regime / Mode (if applicable)
• Subject
• Dates (BC / AD only; no BCE / CE)
• Class: MEM (Memory)
• Last Update
• Wordcount (verified)

Omission, mis-ordering, or unverifiable wordcount INVALIDATES the MEM.

VII. MEM CONNECTION REQUIREMENTS
────────────────────────────────────────────────────────────
Each MEM MUST reference:
• ≥10 other MEM files
• All connections must be to MEM files from the same civilization
• ≥2 connections must be to GEO (geography) MEM files
Each reference MUST explain structural relevance.

CONNECTION TYPE CLASSIFICATION (MANDATORY):
Each MEM connection MUST be classified by relationship type. The
following types are recognized:

A) STRUCTURAL: Similar patterns, organizational forms, or systemic
   relationships (e.g., both MEMs analyze imperial succession patterns)

B) TEMPORAL: Chronological relationships, sequences, or period
   relationships (e.g., MEM–A covers events that precede MEM–B)

C) GEOGRAPHIC: Spatial relationships, location-based connections, or
   regional relationships (e.g., both MEMs focus on Mediterranean region)
   NOTE: GEO MEM files automatically qualify as geographic connections

D) THEMATIC: Similar subjects, topics, or analytical themes
   (e.g., both MEMs analyze military strategy or economic systems)

E) CONTRADICTORY (SCL): Contradictory claims, unresolved tensions, or
   competing interpretations (e.g., MEM–A claims X while MEM–B claims not-X)

Each connection MUST specify its primary type. A connection MAY have
multiple types, but one must be designated as primary.

CONNECTION RELEVANCE (MANDATORY):
Only STRONG connections should be included in the MEM connections list.
Weak or marginal connections do not meet the analytical requirements.

For each connection, the MEM MUST explain:
1. Why this connection matters (relevance to understanding)
2. How it affects interpretation or analysis
3. The nature of the relationship (dependency, parallel, contrast, etc.)
4. Why this connection is STRONG enough to warrant inclusion

BIDIRECTIONAL VALIDATION (REQUIRED):
If MEM–A connects to MEM–B, the system MUST verify:
• MEM–B exists in the repository
• MEM–B can potentially reference MEM–A (same civilization)
• The connection is logically consistent

Authors SHOULD verify target MEMs exist before finalizing connections.
Non-existent MEM references INVALIDATE the connection requirement.

REQUIRED FORMAT:
Each MEM file MUST include a separate "MEM CONNECTIONS" section that
explicitly lists all MEM file connections. This section MUST appear
as a numbered analytical section (e.g., "VII. MEM CONNECTIONS" or
"VIII. MEM CONNECTIONS") and MUST include:

1. A list of ≥10 MEM file references (full file paths or canonical names)
   NOTE: Only STRONG connections should be included. Weak or marginal connections
   do not meet analytical requirements and should not be listed.
2. For each connection:
   a) Connection type (STRUCTURAL / TEMPORAL / GEOGRAPHIC / THEMATIC / CONTRADICTORY)
   b) Explanation of structural relevance and relationship nature
   c) How the connection affects understanding or analysis
   d) Justification for why this is a STRONG connection warranting inclusion
3. Clear identification of which connections are GEO MEM files (≥2 required)
4. Verification that all connections are to MEM files from the same civilization

Example structure:
────────────────────────────────────────────────────────────
VII. MEM CONNECTIONS
────────────────────────────────────────────────────────────

This MEM connects to the following MEM files from [CIVILIZATION]:

1. MEM–[CIV]–[NAME]
   Type: STRUCTURAL
   Relevance: [explanation of structural relevance and relationship nature]
   Impact: [how this connection affects understanding]
   Justification: [why this is a STRONG connection warranting inclusion]

2. MEM–[CIV]–[NAME]
   Type: TEMPORAL
   Relevance: [explanation of chronological relationship]
   Impact: [how temporal sequence affects analysis]
   Justification: [why this is a STRONG connection warranting inclusion]

...

10. MEM–[CIV]–GEO–[LOCATION]
    Type: GEOGRAPHIC
    Relevance: [explanation of spatial/geographic relationship]
    Impact: [how geographic context affects understanding]
    Justification: [why this is a STRONG connection warranting inclusion]

11. MEM–[CIV]–GEO–[LOCATION]
    Type: GEOGRAPHIC
    Relevance: [explanation of spatial/geographic relationship]
    Impact: [how geographic context affects understanding]
    Justification: [why this is a STRONG connection warranting inclusion]

[Continue listing all connections with full specifications]

NOTE: Connection types may overlap (e.g., a GEO MEM connection is both
GEOGRAPHIC and may also be STRUCTURAL or THEMATIC). Designate the
primary type, but acknowledge secondary types when relevant.

────────────────────────────────────────────────────────────
XII-A. COMPLIANCE TIERS (NEW · MANDATORY)
────────────────────────────────────────────────────────────
Every MEM file MUST be classified into one of the following compliance tiers:

DRAFT (Non-Compliant):
• Missing critical requirements (ARC version, wordcount, mandatory clause)
• Structural requirements not met (<8 sections)
• MEM connections insufficient (<10, cross-civilization, or <2 GEO)
• Status: NOT INGESTIBLE · NOT CANONICAL · BLOCKED
• Action Required: Fix critical errors before progression

PARTIAL (Partially Compliant):
• Meets structural requirements (≥8 sections, mandatory clause present)
• Meets some but not all quotation requirements
• MEM connections sufficient but may have minor issues
• ARC version declared but may have parity issues
• Status: INGESTIBLE WITH WARNINGS · NOT CANONICAL · PARTIAL LOCK
• Action Required: Complete missing requirements for full compliance

COMPLIANT (Fully Compliant):
• All mandatory requirements met
• All quotation requirements satisfied
• All MEM connection requirements met (≥10, same civilization, ≥2 GEO)
• ARC version pinned and verified
• Wordcount verified
• Structural compliance confirmed
• Status: INGESTIBLE · CANONIZATION ELIGIBLE · READY FOR LOCK
• Action Required: None — ready for canonical lock

CANONICAL (Locked and Verified):
• All COMPLIANT requirements met
• Status: LOCKED
• Version frozen (additive changes only)
• Wordcount locked
• Content hash verified
• Status: CANONICAL · LOCKED · ADDITIVE UPGRADES ONLY
• Action Required: Cannot modify without unlock

TIER PROGRESSION (STRICT):
DRAFT → PARTIAL → COMPLIANT → CANONICAL

Skipping tiers is PROHIBITED. Each tier must be fully satisfied before
progression to the next.
